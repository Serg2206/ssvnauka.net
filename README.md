# ssvnauka.net

Next.js App Router MVP for Prof. Sergiy Valentinovich Sushkov — surgeon, oncologist, scientist, and educator.

## What changed

- The active site is now a Next.js application with three locales: English, Russian, and Ukrainian.
- Service pages are split into dedicated routes.
- The clinic page includes a secure server intake endpoint with explicit consent and no medical-record storage.
- SEO routes for `robots.txt` and `sitemap.xml` are generated by Next.js.

## Local development

Install dependencies and run the app:

```bash
npm install
npm run dev
```

Then open the local development URL shown by Next.js.

## GitHub Codespaces

This repository includes a dev container config in `.devcontainer/devcontainer.json`.

When you open the project in Codespaces:

1. Dependencies install automatically with `npm ci`.
2. Port `3000` is forwarded automatically.
3. The Next.js app preview opens when the app starts.

Start the app in the Codespace terminal:

```bash
npm run dev -- --hostname 0.0.0.0 --port 3000
```

To use intake/admin features in Codespaces, add repository or Codespace secrets for the environment variables listed below (for example `ADMIN_ACCESS_TOKEN` and `INTAKE_POSTGRES_URL`).

## Environment

Set these variables in Vercel and in your local `.env.local`:

- `NEXT_PUBLIC_SITE_URL` - canonical site URL used for metadata and sitemap generation.
- `ADMIN_ACCESS_TOKEN` - required to open the internal admin intake viewer at `/admin/intake`.
- `INTAKE_POSTGRES_URL` - primary Supabase Postgres connection string for intake storage.
- `INTAKE_POSTGRES_SCHEMA` - schema that holds the intake tables, usually `public`.
- `INTAKE_POSTGRES_REQUESTS_TABLE` - table used for intake requests.
- `INTAKE_POSTGRES_CONSENTS_TABLE` - table used for consent records.
- `INTAKE_POSTGRES_NAME` - label shown in the intake response for the Postgres backend.
- `INTAKE_POLICY_VERSION` - consent/disclaimer policy version stored with each submission.
- `INTAKE_RATE_LIMIT_MAX_REQUESTS` - maximum number of intake submissions allowed in one rate-limit window.
- `INTAKE_RATE_LIMIT_WINDOW_MS` - duration of the intake rate-limit window in milliseconds.
- `INTAKE_DESTINATION_KIND` - label for the live intake backend, usually `private-storage`.
- `INTAKE_DESTINATION_KIND_PREVIEW` - preview-only backend label.
- `INTAKE_WEBHOOK_URL` - optional fallback private storage or CRM webhook endpoint.
- `INTAKE_WEBHOOK_TOKEN` - optional bearer token for the fallback webhook.
- `INTAKE_WEBHOOK_NAME` - optional label shown in the intake response for the fallback webhook.
- `INTAKE_WEBHOOK_URL_PREVIEW` - optional preview-only webhook endpoint.
- `INTAKE_WEBHOOK_TOKEN_PREVIEW` - optional preview-only bearer token.
- `INTAKE_WEBHOOK_NAME_PREVIEW` - optional preview-only label shown in the intake response.

## Notes

- The legacy static HTML files are preserved as source material.
- Real medical data must not be persisted until a private database or storage layer is added.
- The admin intake viewer is server-side only and requires both `ADMIN_ACCESS_TOKEN` and `INTAKE_POSTGRES_URL`.
